# GCREDS (资格与证书服务) 前端白盒化 UI 实现计划

基于微服务团队提供的 `gcreds` 架构意图与 `creds_grpc.pb.go` 接口定义，`gcreds` 主要是作为一个**通用资格管控与证书颁发**的底层引擎。
为了实现“白盒化”（即避免直接修改数据库，通过 UI 完成所有测试和干预），我们需要在 BFF 层（`adminserver` 和 `candidateserver`）实现一套完整的调试与管理 UI。

以下是具体的 UI 规划与接口对接方案：

## 一、 adminserver (管理端) UI 规划

管理端主要负责**资格标准的制定**、**考生资料的审核**、**手动权限干预**以及**自动化证书的模板管理**。建议在侧边栏新增“资格与证书管理”大类：

### 1. 资格定义管理 (Credential Definitions) 【P0 核心重点】
- **UI 描述**: 列表展示系统中所有可用的资格类型（例如：“金融行业2年工作证明”、“CFTA证书”）。支持新建资格模板。
- **对接接口**: 
  - `ListCredentialDefinitions` (获取资格定义列表)
  - `CreateCredentialDefinition` (新建资格标准)

### 2. 资格审核中心 (Application Audit) 【P0 核心重点】
- **UI 描述**: （核心页面）展示所有考生提交的“资格申请”。管理员可根据状态（待审核、已通过、已拒绝）进行过滤。点击某条申请可展开详情（类似站内信详情），查看考生上传的资料，并执行“通过审核(Approve)”、“拒绝(Reject)”或“打回重新上传(Require Resubmit)”操作。
- **对接接口**: 
  - `ListApplications` (分页/条件查询所有考生的申请)
  - `AuditApplication` (提交审核结果，若通过则底层微服务自动下发资格)

### 3. 考生资格人工干预 (Candidate Permissions & Interventions) 【P1 白盒化功能】
- **UI 描述**: 针对单个考生的调试视图。可以手动开通/关闭某考生上传特定资格资料的权限；可以查看考生当前拥有的资格，并支持手动撤销或标记过期。
- **对接接口**: 
  - `GrantUploadPermission` / `RevokeUploadPermission` (手动赋权/撤权，替代 Casdoor enforcer 方案之一)
  - `CheckCandidateQualification` / `GetLatestCredential` (查询资格)
  - `MarkExpired` / `RevokeCredential` (吊销/标记过期)

### 4. 证书与PDF模板管理 (PDF Templates) 【P1 白盒化功能】
- **UI 描述**: 供外部系统（如业务考试系统）颁发自动资格时的 PDF 证书模板配置界面，以及生成的证书记录。
- **对接接口**: 
  - `ListPdfTemplates` / `CreatePdfTemplate` / `UpdatePdfTemplate`
  - `ListPdfRequests` (查看证书生成流水)

---

## 二、 candidateserver (考生端) UI 规划

考生端主要负责**资料的上传申请**以及**已获资格/证书的查看**。

### 1. 我的资格申请 (My Applications)
- **UI 描述**: 考生可以看到自己能够申请的资格列表。点击“申请”后，首先获取上传凭证，上传资料后再提交申请表单。如果被管理员打回，可以在这里重新上传。
- **对接接口**: 
  - `ListCredentialDefinitions` (可申请的资格字典)
  - `RequestUploadUrl` (请求资料上传的 Presigned URL，用于直传对象存储)
  - `SubmitApplication` (正式提交申请)
  - `ListCandidateApplications` (查看自己的申请历史和当前状态)
  - `UpdateApplication` (被要求重新提交时更新资料)

### 2. 我的证书与资格 (My Credentials)
- **UI 描述**: 展示考生已经获得的资格（包括管理员审核通过的，以及考试系统自动发放的 CFTA 证书）。如果附带 PDF，支持下载。
- **对接接口**: 
  - `GetLatestCredential` (拉取当前有效资格)
  - `GetPdfCertificate` (下载 PDF 证书)

## 总结与开发优先级 (Priorities)

结合微服务团队的指示：“你的焦点是管理员 提交 资格模板； 考生提交资格资料； 管理员审核： 通过/拒绝/要求重新提交资料”，同时兼顾“系统白盒化（避免直接改数据库）”的最终目标，后续开发分为两步：

### 【阶段一：P0 核心流程落地】 (首要任务)
1. **优先开发 `adminserver` 的「资格定义管理」和「资格审核中心」**，这是管理员打通主线流程的基石。
2. **接着开发 `candidateserver` 的「我的资格申请」**，跑通 提交->审核->发证 的核心业务闭环。

### 【阶段二：P1 白盒化调试工具包】 (即截图疑问部分)
3. **证书与PDF模板管理**：虽然最终是业务系统 (`gprog`) 调用接口自动发证，但**PDF模板本身需要管理员提前录入到数据库中**。为了白盒化，避免手写 SQL 插入模板，我们需要在 adminserver 提供模板 CRUD 界面。
4. **权限对接与人工干预**：针对考生“是否有资格提交申请”的门槛，微服务团队给出了 Casdoor Enforcer 和 `GrantUploadPermission` 两种方案。为了拥有最全的调试能力，我们会在 Admin 界面实现手动赋权/撤权按钮，作为系统底层的“备用调试手段”。
