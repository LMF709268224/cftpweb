# GLMS 管理端课程配置实现计划

## 1. 当前结论

GLMS 已经在 `adminserver` 和 `candidateserver` 中接入了 gRPC client，环境变量为 `LMS_GRPC_ADDR`，但目前还没有管理端 HTTP BFF 路由、handler 和课程配置页面。

GLMS 的定位应当是“课程内容配置服务”，负责课程本体、资料、章节、课时、前置条件、测验和资源文件。GCC 的 pipeline/unit 负责认证管线编排，其中 `UnitConfig.glms_course_id` 用来引用 GLMS 课程；两者不要混成同一个模型。

## 2. GLMS API 盘点

### 课程 Course

- `CreateCourse`：创建课程，字段包括 `course_id`、`category_id`、`title`、`description`、`thumbnail_object_key`、`duration_min`、`certification_enabled`、`certification_def_id`。
- `UpdateCourse`：更新课程，必须带 `course_id` 和 `version` 做乐观锁。
- `DeleteCourse`：删除课程，必须带 `course_id` 和 `version`。
- `GetCourse`：获取课程基础信息。
- `ListCourses`：按 `category_id`、`published_only` 查询课程列表。
- `PublishCourse` / `UnpublishCourse`：发布/下架课程，必须带 `course_id` 和 `version`。
- `GetCompleteCourse`：获取完整课程结构，包含课程、资料、章节、课时、测验。

注意：`ListCourses` 当前没有分页、关键词、通用状态筛选字段；管理端一期只能暴露“分类”和“仅看已发布”这两个真实筛选能力。分页/关键词/草稿状态筛选需要写 TODO，等 GLMS 补接口。

### 课程资料 CourseMaterial

- 支持 `CreateCourseMaterial`、`UpdateCourseMaterial`、`DeleteCourseMaterial`、`GetCourseMaterial`、`ListCourseMaterials`、`ReorderCourseMaterials`。
- 关键字段：`material_id`、`course_id`、`title`、`material_type`、`description`、`file_object_key`、`file_size`、`sort_order`、`version`。
- `material_type` 枚举：textbook、slides、reference、other。

### 章节与课时 Chapter / Lesson

- 章节支持创建、更新、删除、查询、列表、排序。
- 课时支持创建、更新、删除、查询、列表、排序。
- 课时类型 `lesson_type` 支持 video、text、pdf、image、audio、file、link。
- 课时字段包括 `body`、`media_object_key`、`external_url`、`video_provider`、`video_stream_uid`、`video_embed_code`、`meta_json`。

### 前置条件 Prerequisite

- 支持 `CreatePrerequisite`、`UpdatePrerequisite`、`DeletePrerequisite`、`GetPrerequisite`、`ListPrerequisites`。
- 可配置 required entity 到 target entity 的依赖。
- entity 类型支持 lesson、quiz、chapter、course。
- required result 支持 completed、passed。

### 测验 Quiz / Question / Option

- `Quiz` 支持挂在 lesson、chapter、course 上。
- 测验字段包括 `passing_score`、`time_limit`、`max_attempts`、`allow_retake`、`randomize_questions`、`is_active`。
- 题目类型支持 single choice、multiple choice、true/false。
- 题目和选项都支持 CRUD；题目和选项还支持 reorder。

### 文件对象

- `CreateUploadURL`：生成上传 URL，上传类型包括 course thumbnail、course material、lesson asset、quiz asset。
- `CreateViewURL`：根据 `object_key` 生成临时访问 URL。
- `ListObjects`：按 `prefix`、`page_size`、`page_token` 列文件对象。

文件上传建议流程：前端请求 BFF 创建上传 URL，前端直接 PUT 文件到对象存储，上传成功后把返回的 `object_key` 写入课程/资料/课时/题目字段。

### 报名与进度

- `EnrollCandidateCourse`
- `CompleteLesson`
- `ListCandidateEnrollments`
- `GetCandidateEnrollmentDetail`
- `ListCourseEnrollmentsForAdmin`
- `GetCandidateProgressForAdmin`

这些更偏 candidate 学习流程和管理端查看进度，不建议放在课程配置的一期核心交付里。

### 外部业务绑定

- `BindCourseAssociation`
- `UnbindCourseAssociation`

当 GCC pipeline unit 引用某个 GLMS 课程时，应绑定关联，避免课程被误下架或删除后导致认证管线失效。

## 3. 一期实现范围

一期先做“管理员可维护课程内容”的闭环：

1. 课程列表和基础信息 CRUD。
2. 发布、下架、删除课程，全部带 `version`。
3. 课程资料管理，包含上传文件、预览文件、排序。
4. 章节和课时管理，包含排序。
5. 测验、题目、选项管理，包含排序。
6. 前置条件管理。
7. 完整课程预览，通过 `GetCompleteCourse` 聚合显示。

暂不做：

- candidate 学习进度页面。
- candidate 选课/报名流程改造。
- 课程列表分页、关键词搜索、复杂状态筛选，因为 GLMS 当前 API 不支持。
- pipeline unit 的 GLMS 课程选择器可以放到二期，但一期 BFF 要预留公开的课程列表接口供后续复用。

## 4. 后端 BFF 设计

新增文件：

- `adminserver/handler/lms.go`
- 如 DTO 太多，可补充 `adminserver/handler/lms_schema.go`，保持 GLMS 相关请求/响应集中。

路由统一加在 `adminserver/server/router.go` 的 `/api` 鉴权组内：

```text
GET    /api/lms/courses
POST   /api/lms/courses
GET    /api/lms/courses/{course_id}
GET    /api/lms/courses/{course_id}/complete
PUT    /api/lms/courses/{course_id}
DELETE /api/lms/courses/{course_id}
POST   /api/lms/courses/{course_id}/publish
POST   /api/lms/courses/{course_id}/unpublish

GET    /api/lms/courses/{course_id}/materials
POST   /api/lms/courses/{course_id}/materials
PUT    /api/lms/materials/{material_id}
DELETE /api/lms/materials/{material_id}
POST   /api/lms/courses/{course_id}/materials/reorder

GET    /api/lms/courses/{course_id}/chapters
POST   /api/lms/courses/{course_id}/chapters
PUT    /api/lms/chapters/{chapter_id}
DELETE /api/lms/chapters/{chapter_id}
POST   /api/lms/courses/{course_id}/chapters/reorder

GET    /api/lms/chapters/{chapter_id}/lessons
POST   /api/lms/chapters/{chapter_id}/lessons
PUT    /api/lms/lessons/{lesson_id}
DELETE /api/lms/lessons/{lesson_id}
POST   /api/lms/chapters/{chapter_id}/lessons/reorder

GET    /api/lms/prerequisites
POST   /api/lms/prerequisites
PUT    /api/lms/prerequisites/{prerequisite_id}
DELETE /api/lms/prerequisites/{prerequisite_id}

GET    /api/lms/quizzes
POST   /api/lms/quizzes
PUT    /api/lms/quizzes/{quiz_id}
DELETE /api/lms/quizzes/{quiz_id}

GET    /api/lms/quizzes/{quiz_id}/questions
POST   /api/lms/quizzes/{quiz_id}/questions
PUT    /api/lms/questions/{question_id}
DELETE /api/lms/questions/{question_id}
POST   /api/lms/quizzes/{quiz_id}/questions/reorder

GET    /api/lms/questions/{question_id}/options
POST   /api/lms/questions/{question_id}/options
PUT    /api/lms/options/{option_id}
DELETE /api/lms/options/{option_id}
POST   /api/lms/questions/{question_id}/options/reorder

GET    /api/lms/objects
POST   /api/lms/upload-url
POST   /api/lms/view-url
```

实现原则：

- BFF 只做代理和轻量校验，不操作数据库。
- 所有写操作都校验必填 ID、`version`、枚举值、排序项。
- gRPC 错误统一走现有 `WriteError` / 错误码机制，不把底层英文错误直接展示给前端。
- `ListCourses` 的 query 参数只接 `category_id` 和 `published_only`。分页/关键词/status 参数先不接；如 UI 需要，写 TODO 标注“等待 GLMS ListCourses 补充分页和搜索字段”。
- 不修改 `vendor` 下 protobuf 生成文件。

## 5. 管理端前端设计

新增页面建议：

- `adminserver/web/app/lms/page.tsx`：课程配置首页。
- 后续如页面变重，可拆成 `adminserver/web/app/lms/[courseId]/page.tsx`。

侧边栏增加“课程配置”入口，和当前 “管线配置 / 目录配置” 平级，避免管理员把 GLMS 课程和 GCC 管线混淆。

页面结构建议：

1. 左侧为课程列表：分类筛选、仅看已发布开关、新建课程按钮。
2. 右侧为课程编辑区：基础信息、资料、章节课时、测验、前置条件、预览、发布操作。
3. 课程列表不做假分页；如果数据量多，前端可以本地分组/折叠，但需要明确不是服务端分页。
4. 上传控件根据场景选择 upload type，上传成功后回填 object key。
5. 所有可变文本写入 `adminserver/web/lib/locales/zh.ts` 和 `en.ts`，页面中通过 `useTranslation()` 使用。
6. API 请求统一使用 `apiClient`。
7. 时间字段统一使用 `formatBackendDate`。
8. 新增 TSX 页面显式 `import React`，并包裹标准 `<Sidebar />` 布局。

## 6. 与 GCC / pipeline 的衔接

二期建议在管线配置里补一个 GLMS 课程选择器：

- 编辑 GCC `UnitConfig` 时，如果 `has_learning=true`，允许选择已发布的 GLMS 课程。
- 保存 pipeline structure 时把课程 ID 写入 `glms_course_id`。
- 同步调用 `BindCourseAssociation(course_id, "gcc_unit", unit_id)`。
- 更换或清空课程时调用 `UnbindCourseAssociation`。

需要确认：绑定实体的 `entity_type` 字符串是否已有下游约定。暂定用 `gcc_unit`，实现前最好和 GLMS/GCC 维护方确认。

## 7. candidate 侧后续计划

candidate 侧当前仍以 GCC/GPROG pipeline 为主。GLMS 接进来后建议分两步：

1. 在课程详情/学习页使用 `GetCompleteCourse` 展示课程内容。
2. 学习行为使用 `EnrollCandidateCourse`、`CompleteLesson`、`ListCandidateEnrollments`、`GetCandidateEnrollmentDetail` 更新学习进度。

这些不要和管理端课程配置一期混做，避免影响购买、资格、考试流程。

## 8. 需要确认的问题

1. `course_id`、`material_id`、`chapter_id`、`lesson_id`、`quiz_id` 等 ID 是由前端/BFF 生成，还是 GLMS 服务端可为空自动生成？当前 proto 的 create request 都带 ID 字段，但 response 也返回 ID，需要确认真实约定。
2. 删除是否为软删除？已发布或已绑定课程删除/下架时 GLMS 是否会拦截？
3. `ListCourses` 是否会补分页、关键词和状态筛选？如果不会，管理端列表需要接受只按分类/发布状态筛选。
4. `CreateUploadURL` 返回的 URL 是否要求特定 HTTP method/header？前端上传时是否需要携带 content-type。
5. `meta_json`、`media_items_json` 的结构是否已有业务约定？如果没有，一期先提供结构化表单生成 JSON，避免让管理员手写。
6. `certification_def_id` 应该来自 GCREDS 的证书/资格定义，还是管理员手填？建议用下拉选择，不允许手写。

## 9. 建议实施顺序

1. 后端先补 `/api/lms/*` BFF，完成课程、资料、章节、课时的最小闭环。
2. 再补测验、题目、选项、前置条件和完整课程预览。
3. 前端实现课程配置页，先让管理员能完整维护一门课程。
4. 接入对象上传和预览。
5. 加入发布/下架和版本冲突提示。
6. 管线配置页再接 GLMS 课程选择器与绑定关系。
7. 最后再处理 candidate 侧学习页和进度。

## 10. 确认更新

根据目前确认结果，后续实现按下面约束执行：

1. 所有 GLMS 新建对象 ID 都由 `adminserver` 生成 ULID，包括 `course_id`、`material_id`、`chapter_id`、`lesson_id`、`prerequisite_id`、`quiz_id`、`question_id`、`option_id`。实现时优先沿用项目现有 `github.com/oklog/ulid/v2` 的生成方式。
2. 删除操作直接调用 GLMS 对应删除接口即可，BFF 不需要判断下游是软删除还是硬删除；BFF 只负责传 `id` 和 `version`，并把错误码规范化返回给前端。
3. `ListCourses` 后续会补分页、关键词和状态筛选。当前实现可以先只接已有 `category_id`、`published_only`；涉及分页/关键词/status 的 UI 和 BFF 参数保留 TODO，等 GLMS proto 更新后接入。
4. 文件上传必须认真处理 `content_type`：
   - GLMS 服务端当前会把 `content_type` 传入 S3 `PresignPutObject` 的 `ContentType` 字段。
   - 如果生成 presigned URL 时带了 `Content-Type`，前端真正 PUT 文件时最好使用完全相同的 `Content-Type` header，否则可能出现 S3 签名不匹配，或对象存储里的文件 MIME 类型不正确，进而影响浏览器预览、下载文件名/类型识别。
   - 前端请求上传 URL 时建议传 `file.type || "application/octet-stream"`，随后 `fetch(upload_url, { method: "PUT", headers: { "Content-Type": 同一个值 }, body: file })`。
5. 已查看下游仓库 `github.com/afnandelfin620-star/cftptest` 的 GLMS 服务端实现：
   - `glms/server/grpc_lesson.go` 中 `meta_json` 只校验是否为合法 JSON，没有更细 schema。
   - `glms/server/grpc_quiz_question.go` 中 `media_items_json` 也只校验是否为合法 JSON，没有更细 schema。
   - 课时字段有类型约束：text 课时必须有 `body`；link 课时必须有合法 `external_url`；video/pdf/image/audio/file 等媒体类课时必须有合法 `media_object_key`。
   - 因此一期前端不要让管理员手写 JSON。建议提供结构化表单，默认生成 `{}` 或 `[]`，只在需要扩展媒体信息时生成合法 JSON 字符串。
6. 能下拉选择的字段都用下拉选择，不让管理员手写：
   - 枚举类：`material_type`、`lesson_type`、`entity_type`、`required_result`、`quizzable_type`、`question_type`、`upload_type`。
   - 关联对象：`category_id`、`certification_def_id`、`course_id`、`chapter_id`、`lesson_id`、`quiz_id`、`question_id` 等，优先通过现有列表接口加载选项。
   - 如果某个下拉数据源暂时没有列表接口，则在代码里写 TODO 标明等待下游接口或复用现有服务接口，避免退回到长期手填。
