   - 本次补充：`@embedpdf/vue-pdf-viewer` 运行时 import 的 `@embedpdf/snippet` 必须固化为项目直接依赖，并纳入 Dockerfile resolve check，不能只检查外层包。
   - 本次补充：`vue-router` 生产构建拆出的 `dist/devtools-*.mjs` 会直接 import `@vue/devtools-api`，仅 `require.resolve("@vue/devtools-api")` 通过不代表 Rollup 一定能解析；必须在 Vite alias 中指向 `node_modules/@vue/devtools-api/dist/index.js`，并在 Dockerfile 检查该实际文件。

   - 本次补充：Vue 运行时内部包 `@vue/reactivity` / `@vue/runtime-core` / `@vue/runtime-dom` / `@vue/shared` 会被 Vue ESM bundler 文件裸导入，生产构建不能只依赖 `vue` 的间接依赖；这些包必须固定为项目直接依赖，并在 Vite alias 中指向对应 `dist/*esm-bundler.js` 文件。

## 后续必须遵守的规则

1. 所有业务 API 必须通过统一 `apiClient`
   - candidate 使用 `candidateserver/vue-web/src/lib/apiClient.ts`。
   - admin 使用 `adminserver/web/lib/apiClient.ts`。
   - 不允许在页面里新增裸 `fetch("/api/...")`，除非是非 JSON 文件流或对象存储直传。

1.1. 所有前端构建依赖必须写入 package 和 lockfile
   - 禁止在 Dockerfile、CI 脚本、部署脚本里使用 `npm install --no-save` / `yarn add` 临时补依赖。
   - 新增依赖必须先改 `package.json`，再提交对应 lockfile。
   - Docker 构建阶段只能执行 `npm ci && npm run build`，不能在 `npm ci` 后修改依赖树。
   - 如果 Rollup 报 `failed to resolve import`，优先检查依赖是否进入 lockfile，而不是继续在 Dockerfile 里补临时安装。
   - Dockerfile 必须保留前端依赖 resolve 检查，至少覆盖直接依赖和已知容易缺失的 transitive dependency。
   - 对会被 Rollup 直接解析的运行时 transitive dependency，例如 `@embedpdf/snippet`，必须提升为项目直接依赖或明确纳入 resolve check，不能只依赖外层包的间接声明。
   - 对 Vue ESM bundler 链路中的内部包，必须保持 `vue`、`@vue/runtime-core`、`@vue/runtime-dom`、`@vue/reactivity`、`@vue/shared` 版本一致，并提交 `package-lock.json`。
   - 对 Vue Devtools 链路中的内部包，必须保持 `@vue/devtools-api`、`@vue/devtools-kit`、`@vue/devtools-shared` 版本一致，并提交 `package-lock.json`。
   - 如果包存在但 Rollup 仍报 `failed to resolve import`，需要检查 Vite/Rollup 的 ESM 解析路径；必要时在 `vite.config.ts` 添加稳定 alias，并把 alias 指向的实际文件加入 Dockerfile 检查。

2. 必须使用裸 `fetch` 时，需要同时满足三点
   - 有 `AbortController` 超时。
   - 有 catch/finally，确保 loading 一定能结束。
   - 有用户可见失败态，例如 toast、错误卡片、重试按钮或返回按钮。

2.1. 集合 API 路由必须兼容有无尾斜杠
   - 前端列表接口统一优先请求无尾斜杠路径，例如 `/api/orders`，后端必须注册对应 exact route。
   - 如果后端使用 chi 的 `r.Route("/xxx")` + `r.Get("/")`，必须额外注册 `r.Get("/xxx", sameHandler)`。
   - 不能用一个过宽的 `/api/pipeline` 路由组去包 PDF 长超时接口，否则会吞掉 `/api/pipeline/courses/.../complete` 等普通业务接口。

3. PDF/文件下载接口必须设置文件名
   - 后端必须设置 `Content-Type`。
   - 下载用 `Content-Disposition: attachment; filename="xxx.ext"`。
   - 预览用 `Content-Disposition: inline; filename="xxx.ext"`。
   - 前端 blob 下载也要解析 `Content-Disposition`，并提供本地兜底文件名。

4. 复杂组件必须可降级
   - PDF viewer、支付组件、第三方嵌入页等不能成为单点白屏源。
   - 组件失败时必须有 fallback UI，例如 iframe、普通链接、重试按钮。
   - 大 PDF 预览不能默认走“整文件下载到内存后再渲染”的组件链路；课程 PDF 优先使用浏览器原生 PDF iframe 预览，让浏览器通过 Range/206 分片读取。
   - 如果继续使用第三方 PDF viewer，必须先确认它是否支持 Range streaming；如果内部调用 `arrayBuffer()`，则只适合小文件或作为可选增强，不适合作为 50MB+ 课件的默认预览方式。
   - 预览代理接口必须显式支持 `HEAD`、透传 `Range`/`If-Range`，并设置 `Accept-Ranges: bytes`、`Content-Disposition: inline`。
   - 签名预览 URL 有效期不能过短；浏览器 PDF 预览会在用户翻页时继续请求后续分片，过期会导致中途 401。

5. 页面 loading 不能无限持续
   - 所有 `loading = true` 必须有 `finally` 复位。
   - 初始加载页必须有错误态或空态，不能只有 `Loading...`。
   - PDF 预览必须在新页面初始阶段立刻显示自定义 loading，不能只依赖第三方 viewer 内部 loading。
   - 慢加载超过阈值时必须显示用户可操作入口，例如重试和“直接打开”。
   - 所有加载态必须有用户可读文字说明，不能只显示 spinner，也不能出现白板让用户不知道系统在做什么。
   - 打开外部第三方页面前必须优先进入本站中转/加载页，展示“正在打开/正在加载”的文字、失败态和重试/返回入口；不能直接让用户面对第三方白屏。
   - 新增或改动任何 `window.open` / `window.location.href` 外部跳转时，都必须检查是否需要中转加载页或页面内显式提示。

## 建议增加的自动检查

1. lint 规则
   - 禁止页面层直接调用 `fetch("/api/")`。
   - 允许白名单：`apiClient.ts`、对象存储上传、文件流下载。
   - 禁止 Dockerfile/CI 中出现 `npm install --no-save`。

2. PR checklist
   - 新增 API 请求是否走统一 `apiClient`？
   - 新增文件下载是否有 `Content-Disposition` 文件名？
   - 新增 loading 是否有失败态和 finally？
   - 新增 loading 是否有明确文字说明，而不是只有图标或白板？
   - 新增外部跳转是否有本站加载提示、中转页或失败兜底？
   - 新增第三方组件是否有 ErrorBoundary 或 fallback？

3. Smoke test
   - 模拟 API 15 秒无响应，页面必须显示超时而不是一直 loading。
   - 模拟 PDF 预览失败，页面必须能返回课程页。
   - 下载发票后文件名必须以 `.pdf` 结尾。
   - 访问 candidate 关键集合接口时，无尾斜杠路径必须返回 200 或业务状态，不允许 404：`/api/pipeline`、`/api/mall/pipelines`、`/api/orders`、`/api/messages`、`/api/progress`。
   - 点击发票、考试排期、证书外链等外部跳转入口后，必须立即看到本站文字加载提示，不能只出现第三方白屏。
