## 本次已修复的问题

1. 支付页缺少测试卡提示
   - 问题：Stripe 测试环境没有明确提示测试卡号，测试人员不知道如何完成支付。
   - 修复：candidate 购买弹窗在 Stripe 支付方式和 embedded checkout 区域展示测试卡 `4242 4242 4242 4242`，任意有效日期/CVV。
   - 文件：`candidateserver/vue-web/src/components/PurchaseDialog.vue`

2. 页面/API 卡死
   - 问题：接口无响应或 PDF 预览组件异常时，页面可能一直停留在 loading 或白屏。
   - 修复：
     - candidate `apiClient` 默认 15 秒超时，网络/超时统一 toast。
     - admin `apiClient` 默认 15 秒超时，网络/超时统一 toast。
     - candidate 顶层 `App.vue` 增加 ErrorBoundary 兜底 UI。
     - PDF 预览页加载 PDF 默认 120 秒超时，embedpdf 组件异常时降级到 iframe 预览。
   - 文件：
     - `candidateserver/vue-web/src/lib/apiClient.ts`
     - `adminserver/web/lib/apiClient.ts`
     - `candidateserver/vue-web/src/App.vue`
     - `candidateserver/vue-web/src/pages/PdfPreviewPage.vue`

3. 发票 PDF 下载丢失后缀名
   - 问题：订单页下载发票时浏览器保存成无 `.pdf` 后缀的随机文件名。
   - 修复：
     - candidate 后端 `/api/invoices/{orderId}/pdf` 代理 Stripe PDF，强制设置 `Content-Disposition: attachment; filename="xxx.pdf"`。
     - candidate 订单页使用该 PDF 接口下载 blob，并解析响应头文件名；失败时兜底为 `invoice-{orderId}.pdf`。
   - 文件：
     - `candidateserver/handler/invoice.go`
     - `candidateserver/vue-web/src/pages/OrdersPage.vue`

4. Docker 构建中临时安装前端依赖导致依赖树被破坏
   - 问题：Dockerfile 在 `npm ci` 后执行 `npm install --no-save @embedpdf/vue-pdf-viewer@2.14.4`，会改变 `node_modules` 依赖树但不更新 lockfile，反复引发 Rollup 无法解析 `@vue/reactivity` / `@vue/devtools-api` / `@embedpdf/vue-pdf-viewer` 等依赖。
   - 修复：删除 Dockerfile 中的 `npm install --no-save`，所有前端构建依赖必须固化在 `package.json` 和 `package-lock.json`。
   - 文件：`candidateserver/Dockerfile`
   - 重复记录：该类问题已经至少以 `@vue/reactivity`、`@embedpdf/vue-pdf-viewer`、`@vue/devtools-api` 三种形式重复出现，后续一律按“依赖固化问题”处理，不允许再通过 Dockerfile 临时安装绕过。

## 后续必须遵守的规则

1. 所有业务 API 必须通过统一 `apiClient`
   - candidate 使用 `candidateserver/vue-web/src/lib/apiClient.ts`。
   - admin 使用 `adminserver/web/lib/apiClient.ts`。
   - 不允许在页面里新增裸 `fetch("/api/...")`，除非是非 JSON 文件流或对象存储直传。

1.1. 所有前端构建依赖必须写入 package 和 lockfile
   - 禁止在 Dockerfile、CI 脚本、部署脚本里使用 `npm install --no-save` / `yarn add` 临时补依赖。
   - 新增依赖必须先改 `package.json`，再提交对应 lockfile。
   - Docker 构建阶段只能执行 `npm ci && npm run build`，不能在 `npm ci` 后修改依赖树。
   - 如果 Rollup 报 `failed to resolve import`，优先检查依赖是否进入 lockfile，而不是继续在 Dockerfile 里补临时安装。
   - Dockerfile 必须保留前端依赖 resolve 检查，至少覆盖直接依赖和已知容易缺失的 transitive dependency。

2. 必须使用裸 `fetch` 时，需要同时满足三点
   - 有 `AbortController` 超时。
   - 有 catch/finally，确保 loading 一定能结束。
   - 有用户可见失败态，例如 toast、错误卡片、重试按钮或返回按钮。

3. PDF/文件下载接口必须设置文件名
   - 后端必须设置 `Content-Type`。
   - 下载用 `Content-Disposition: attachment; filename="xxx.ext"`。
   - 预览用 `Content-Disposition: inline; filename="xxx.ext"`。
   - 前端 blob 下载也要解析 `Content-Disposition`，并提供本地兜底文件名。

4. 复杂组件必须可降级
   - PDF viewer、支付组件、第三方嵌入页等不能成为单点白屏源。
   - 组件失败时必须有 fallback UI，例如 iframe、普通链接、重试按钮。

5. 页面 loading 不能无限持续
   - 所有 `loading = true` 必须有 `finally` 复位。
   - 初始加载页必须有错误态或空态，不能只有 `Loading...`。

## 建议增加的自动检查

1. lint 规则
   - 禁止页面层直接调用 `fetch("/api/")`。
   - 允许白名单：`apiClient.ts`、对象存储上传、文件流下载。
   - 禁止 Dockerfile/CI 中出现 `npm install --no-save`。

2. PR checklist
   - 新增 API 请求是否走统一 `apiClient`？
   - 新增文件下载是否有 `Content-Disposition` 文件名？
   - 新增 loading 是否有失败态和 finally？
   - 新增第三方组件是否有 ErrorBoundary 或 fallback？

3. Smoke test
   - 模拟 API 15 秒无响应，页面必须显示超时而不是一直 loading。
   - 模拟 PDF 预览失败，页面必须能返回课程页。
   - 下载发票后文件名必须以 `.pdf` 结尾。
