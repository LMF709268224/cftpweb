# CFTP 系统环境与运维部署指南

本文档面向系统运维与后端部署人员，详细说明如何从零开始搭建一套完整的 CFTP（认证培训平台）环境。系统包含前端、BFF 网关以及 10 多个底层核心微服务。

---

## 1. 基础设施准备 (Infrastructure)

在部署系统之前，需要准备好以下基础运行环境及组件：

### 1.1 服务器与操作系统
* **操作系统**：推荐使用 Ubuntu 22.04 LTS 或 Debian 12。
* **资源配置推荐 (生产/类生产环境)**：
  * **数据库节点** (PostgreSQL/Redis)：4核 16G 内存。
  * **中间件节点** (NATS/Casdoor/MinIO)：4核 8G 内存。
  * **微服务集群节点** (Go/Node.js)：推荐使用 Kubernetes 集群（至少 3 个 Worker 节点，单节点 4核 8G起步）。

### 1.2 核心中间件与依赖组件
所有服务强依赖以下四层核心基础设施，请在启动任何微服务前确保它们已就绪：

1. **关系型数据库 (PostgreSQL 14+)**
   * **作用**：存储所有微服务的业务核心数据。
   * **部署位置**：建议内网托管 RDS 或自建高可用主从集群。
2. **缓存与分布式锁 (Redis 7.0+)**
   * **作用**：存放防重放 Token、BFF 缓存、会话管理、分布式锁。
3. **消息队列事件总线 (NATS JetStream)**
   * **作用**：用于微服务间的异步解耦。例如支付成功后通过 NATS 发布 `OrderPaid` 事件，`gprog` 监听到事件后自动下发进度。
4. **对象存储服务 (AWS S3 / MinIO / Aliyun OSS)**
   * **作用**：存储考生上传的文件、视频、课件以及生成的 PDF 证书。
5. **集中式身份认证 (Casdoor)**
   * **作用**：提供统一的 OAuth2/OIDC 单点登录、RBAC 角色权限管理和 JWT 鉴权分发。

## 2. 系统的服务拓扑清单

整个 CFTP 平台分为 **Web 前端层**、**BFF 网关层** 和 **后台 gRPC 微服务层**。

### 2.1 Web 前端与 BFF (所在仓库：`cftpweb`)
分为面向考生的 `cand` 端和面向内部运营的 `admin` 端。

* **candweb**：考生前端（Vue3 + Vite）。需通过 Nginx 托管静态资源。
* **candbff**：考生网关（Node.js / Go）。对外暴露 REST API，校验 JWT，向内调用 gRPC。
* **adminweb**：管理后台前端（Vue3 + Vite）。
* **adminbff**：管理网关。处理管理端复杂的聚合逻辑。

### 2.2 核心后端微服务 (所在仓库：`cftptest`)
共有 16 个根据业务领域划分的 Go 微服务，内部全部使用 gRPC 通信。在部署新环境时，需将以下微服务全部启动：

| 服务名称 | 服务标识 | 职责描述 |
| :--- | :--- | :--- |
| **商品目录服务** | `gcc` (Course Catalog) | 管理所有管线、课程单元与价格（历史数据），配合商城服务使用。 |
| **认证配置服务** | `gcreds` (Credentials) | 定义认证大纲、所需条件及证书模版配置。 |
| **外部考试网关** | `geemock` / `geex` | 对接第三方考试机构（如 Prometric、GEE），处理模拟和真实的成绩回调。 |
| **考试管理服务** | `gexam` (Exam) | 处理考生的考试预约、排期与成绩录入。 |
| **学习管理系统** | `glms` (LMS) | 管理章节、课时，并接收来自视频播放器的进度上报同步。 |
| **商城系统** | `gmall` (Mall) | 统一的组合包（Bundle）购买、价格快照及订单生成。 |
| **会员系统** | `gmbr` (Membership) | 会员等级控制、自动续费周期管理。 |
| **分布式发号器** | `gmid` (ID Gen) | 提供高性能的分布式自增序列号。 |
| **消息通知服务** | `gmsg` / `gmail` | 负责触发站内信以及对接 SMTP 发送外部邮件。 |
| **支付服务** | `gpay` (Pay) / `gstripe` | 发起支付会话、拉取发票，接收 Stripe Webhook 完成支付闭环。 |
| **PDF 生成器** | `gpdf` (PDF) | 使用无头浏览器或模板引擎后台生成正式的成绩单与证书。 |
| **进度大管家** | `gprog` (Progress) | 跨域协同引擎，整合上述所有状态，计算考生的管线大进度和解锁逻辑。 |

## 3. 环境初始化与数据预备 (Initialization)

部署代码只是第一步，新环境能够跑起来必须要有数据初始化动作：

### 3.1 数据库结构与配置 (DB Schema)
* 大部分 Go 微服务基于 `gorm` 会在启动时通过 `AutoMigrate` 自动建表，但建议在 CI/CD 流水线中统一收口，执行专门的 DDL 迁移脚本。
* **数据来源**：系统级字典表（例如语言包、枚举、系统内置角色权限配置集），需要通过跑对应的 `seed.sql` 脚本导入初始数据。

### 3.2 Casdoor 认证初始化 (Identity & SSO)
在启动 `cftpweb` 的 BFF 之前，必须在 Casdoor 控制台中手动或通过 API 进行如下设定：
1. **创建 Organization（组织）**：例如创建名为 `cftp-org`。
2. **创建 Application（应用）**：
   * 创建 `CFTP-Cand` 和 `CFTP-Admin` 两个应用。
   * 配置回调 URL (Redirect URIs)，指向前端部署的域名。
   * **记录 `Client ID` 和 `Client Secret`**，将它们填入 `candbff` 和 `adminbff` 的环境变量中。
3. **初始化内置角色 (Roles)**：
   * 在 Casdoor 中必须预先创建诸如 `admin`, `member-affiliate`, `candidate-basic` 等角色，供 `gmbr` (会员服务) 自动分配权限时使用。

### 3.3 第三方服务 API Key 初始化
在 Kubernetes Secrets 或 `.env` 文件中，必须提前挂载好以下外部密匙：
* **Stripe**：`STRIPE_SECRET_KEY` 及对应的 `WEBHOOK_SECRET`（保证支付回调验签）。
* **SMTP**：邮件发送账号与密码。
* **OSS/S3**：`AWS_ACCESS_KEY_ID` 和 `AWS_SECRET_ACCESS_KEY`，且需要在对象存储上预先建好公开可读的桶（如 `cftp-public`）。

## 4. 配置文件要求示例

所有微服务通常支持读取环境变量或 `config.yaml`。启动服务时最基本的配置要素包括：

```yaml
# 典型微服务基础配置
Server:
  gRPC_Port: 9001
  HTTP_Port: 8001 (如适用用于健康检查)

Database:
  DSN: "postgres://user:pass@db-host:5432/cftp_db?sslmode=disable"

Redis:
  Addr: "redis-host:6379"

Nats:
  URL: "nats://nats-host:4222"

Registry:
  # 如果使用 Consul 做服务发现，需配置地址
  Address: "consul-host:8500"
```
BFF 配置额外需要包含上述的 Casdoor Endpoint 及 Token、Client ID 等认证材料。

---

> [!TIP]
> 部署顺序建议：
> `DB / Redis / NATS` -> `Casdoor (SSO)` -> `gmid (基础序列号服务)` -> `其余 gRPC 后端微服务` -> `adminbff / candbff` -> `Vue 前端 Nginx 部署`
