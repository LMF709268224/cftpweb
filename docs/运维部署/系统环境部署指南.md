# CFTP 系统运维架构与新环境部署指南

本文档为 CFTP 认证培训平台（包含后台 `admin` 体系与前台 `cand` 体系及所有底层 Go 微服务）的全新环境部署指南。通过本文档，运维人员可以清晰地了解在部署一套全新环境时，需要准备什么机器配置、安装什么基础设施环境、微服务清单，以及最为关键的数据初始化流转。

---

## 1. 部署架构与机器准备

### 1.1 架构拓扑 (运维视角)
整个系统分为以下几个层级：
1. **网关接入层**：Traefik (暴露 HTTPS，处理 Ingress)。
2. **前端与 BFF 层**：
   - 考生端：`candweb` (前端静态文件) + `candbff` (Go网关，聚合请求)
   - 管理端：`adminweb` (前端静态文件) + `adminbff` (Go网关，聚合请求)
3. **认证层**：Casdoor (单点登录与鉴权中心)。
4. **底层微服务群 (Go)**：负责各个业务域，共 13+ 个服务，基于 gRPC 通信。
5. **基础设施层**：PostgreSQL, Redis, NATS, MinIO, Grafana/Loki。

### 1.2 推荐机器配置与环境要求
如果是一套全新的**测试或轻量级准生产环境**，推荐以下配置规划：

| 节点类型 | 数量 | 推荐配置 | 部署内容 |
| :--- | :--- | :--- | :--- |
| **K8s Master / LB** | 1台 | 4核 8G, 100G 磁盘 | K3s/K8s master, Traefik, Ingress 控制器 |
| **K8s Worker Node** | 2台 | 8核 16G, 200G 磁盘 | Casdoor, BFF, Go微服务集群, NATS |
| **Database Server** | 1台 | 8核 16G, 500G 磁盘 | PostgreSQL (核心业务数据), Redis (缓存与分布式锁) |
| **Storage Server** | 1台 | 4核 8G, 1TB+ 磁盘 | MinIO (对象存储), Loki (日志存储) |

> [!TIP]
> 对于高可用生产环境，建议 PostgreSQL 和 Redis 剥离出来，使用云厂商托管服务（如阿里云 RDS/Redis），MinIO 替换为云原生 S3/OSS。

**基础软件环境：**
- Kubernetes 1.25+ 或 K3s
- 容器运行时：Containerd / Docker
- 数据库：PostgreSQL 14+ 
- 缓存：Redis 6.0+

## 2. 需部署的服务清单与配置说明

### 2.1 基础设施组件
- **NATS JetStream**：服务间异步消息总线（必须开启持久化存储 PVC，端口 4222）。
- **MinIO**：兼容 S3 的对象存储，提供图片、视频、PDF 文件存放。
- **Traefik / Ingress**：处理外部 HTTPS 流量与域名路由。
- **Vector / Loki / Grafana**：日志采集与检索可视化大盘。

### 2.2 核心支撑服务
1. **Casdoor**：身份认证中心（需配置数据库连接）。
2. **cfgserver**：配置中心（所有业务微服务启动前，必须先启动此服务，统一提供各微服务的配置文件与连接字符串分发）。

### 2.3 业务 BFF 层 (网关)
- **adminbff**：管理端接口网关，直连微服务，依赖 Casdoor 鉴权。
- **candbff**：考生端接口网关，直连微服务，依赖 Casdoor 鉴权。

### 2.4 底层 Go 微服务集群 (共计 13 个核心服务)
所有微服务的部署清单（Kubernetes YAML）均位于 `cftptest/deployment/` 目录下，部署清单如下：
- `gcc` (课程与认证目录，最为核心)
- `glms` (学习管理资源，课件与视频)
- `gprog` (进度引擎，全链路协同)
- `gmall` (商城与订单生成)
- `gpay` (支付网关，需配置 Stripe 密钥)
- `gmbr` (会员体系)
- `gcreds` (资格与证书发放)
- `gexam` (考试服务，需配置 GEE/Prometric 参数)
- `gmid` (身份映射，Casdoor UUID 与系统 ULID 相互映射)
- `gmsg` (站内信与系统消息通知)
- `gmail` (邮件服务，需配置 SMTP 参数)
- `gpdf` (PDF 生成服务，发证依赖)
- `gaudit` (审计服务)

## 3. 数据初始化流程 (最关键步骤)

在微服务启动前，必须完成数据库表的建立和部分基础数据的初始化。所有的初始化脚本都保存在后端仓库的 `sqlscripts` 文件夹中。

### 3.1 建库与建表顺序
因为系统采用微服务架构，推荐的做法是在同一个 PostgreSQL 实例中，**为每个服务创建独立的 Schema 或独立的 Database**（如 `gcc_db`, `gprog_db` 等），以保证业务数据隔离。

**执行命令示例 (需 DBA 或运维执行)：**
每个微服务代码包的 `sqlscripts/` 目录下都有两类文件：
1. `postgres_tables.sql`：建表结构语句。
2. `postgres_{servicename}.sql`：初始化数据语句。

```bash
# 1. 审计服务数据初始化
psql -d gaudit_db -f gaudit/sqlscripts/postgres_tables.sql
psql -d gaudit_db -f gaudit/sqlscripts/postgres_audit.sql

# 2. 目录服务数据初始化 (系统基石)
psql -d gcc_db -f gcc/sqlscripts/postgres_tables.sql
psql -d gcc_db -f gcc/sqlscripts/postgres_gcc.sql

# 后续依次对 glms, gmail, gmall, gmbr, gmid, gmsg, gpay, gprog, gcreds, gexam 进行相同的建表与初始化操作。
```

### 3.2 业务种子数据来源
在数据库建表完成后，除了上面跑的 sql 脚本，系统要完全转起来还需要一些额外的动作：

1. **Casdoor 初始化**：
   - 需要在 Casdoor 网页端初始化一个 `CFTP` 组织 (Organization)。
   - 初始化两个应用 (Application)：一个用于 `adminweb` 登录，一个用于 `candweb` 登录。获取 `Client ID` 与 `Client Secret` 后交由 `cfgserver` 或 BFF 配置。
   - 配置初始的超级管理员账号。

2. **第三方密钥配置录入**：
   - 支付平台 (Stripe) 的 Webhook Secret 与 API Key。
   - 邮件服务器 (SMTP) 的发信账号、密码与端口。
   - 外部考试系统 (Prometric/GEE) 的通讯凭证。
   - 这些配置统一写入 `cfgserver` 中，微服务启动时将自动拉取。

> [!TIP]
> 部署顺序黄金准则：
> `PostgreSQL / Redis / NATS` -> `Casdoor` -> `cfgserver` -> `gmid (基础序列号服务)` -> `gcc (核心目录)` -> `其余 11 个 gRPC 后端微服务` -> `adminbff / candbff` -> `Vue 前端部署`
