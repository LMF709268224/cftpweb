# CFTP 系统架构与部署说明

## 1. 项目概述

CFTP 是一个基于微服务架构的认证培训平台，主要用于管理认证项目、考生学习进度、考试预约、支付订单、消息通知以及证书发放等业务流程。

本仓库主要包含 CFTP 后端系统，包括 Go 微服务、共享协议与工具包、Kubernetes 部署清单、测试工具以及相关配置脚本。

整个系统的核心特点如下：

- 后端采用 Go 微服务拆分。
- 服务之间主要通过 gRPC 通信。
- 前端请求统一经过 BFF/API Gateway 进入后端。
- 用户认证集成 Casdoor。
- 持久化数据主要使用 PostgreSQL。
- Redis 用于缓存和分布式锁。
- NATS JetStream 用于服务之间的异步事件通信。
- 对象存储用于保存 PDF、课程资料、视频等文件。
- Stripe、SMTP、Prometric/GEE 等外部系统提供支付、邮件、考试等能力。

---

## 2. 整体系统架构

系统整体访问链路可以理解为：

```text
考生端 Web / 管理端 Web
        |
        v
BFF Gateway / API Gateway
        |
        | 认证：Casdoor / JWT
        |
        v
后端 Go 微服务集群
        |
        +--> PostgreSQL：业务数据存储
        +--> Redis：缓存、分布式锁
        +--> NATS JetStream：异步消息、事件驱动
        +--> S3 / MinIO / OSS：课程资料、PDF、视频等对象存储
        +--> SMTP：邮件发送
        +--> Stripe：支付
        +--> Prometric / GEE：考试预约与成绩回传
```

从业务角度看，前端不会直接访问各个微服务，而是先访问 BFF 层。BFF 负责鉴权、聚合数据、调用后端 gRPC 服务，然后把结果返回给前端。

---

## 3. 核心访问流程

### 3.1 前端访问

系统面向两类主要用户：

- 考生端用户：购买课程、学习课程、报名考试、查看进度、查看证书。
- 管理端用户：管理课程、认证项目、订单、考试、证书、消息等。

这些用户通过 Web 前端访问系统。

### 3.2 BFF / API Gateway

BFF 是前端和后端微服务之间的入口层，主要职责包括：

- 接收前端 HTTP 请求。
- 校验用户登录状态和 JWT Token。
- 对接 Casdoor 完成身份认证。
- 根据业务场景调用一个或多个后端 gRPC 服务。
- 聚合多个服务的数据，返回适合前端使用的响应结构。
- 管理后端 gRPC 客户端连接。

可以把 BFF 理解成“前端专用后端”。它不负责全部核心业务逻辑，而是负责请求编排和数据聚合。

### 3.3 后端微服务

后端按照业务领域拆分成多个 Go 微服务。每个微服务负责相对独立的业务边界，并通过 gRPC 对外提供能力。

微服务之间既有同步调用，也有异步事件通知。

同步调用一般使用 gRPC，例如：

- 商城服务查询课程目录。
- 支付服务创建支付会话。
- 进度服务查询考生学习状态。

异步事件一般通过 NATS JetStream，例如：

- 支付成功后触发订单履约。
- 课程进度变化后推动认证流程。
- 需要发送邮件或站内信时发布通知事件。

## 4. 主要微服务说明

### 4.1 cfgserver：配置中心服务

`cfgserver` 是系统配置中心，负责向各个微服务提供运行时配置。

主要职责：

- 管理各服务启动所需配置。
- 为微服务下发数据库、Redis、NATS、外部系统等连接配置。
- 支持服务启动时拉取配置。
- 可作为集中配置管理入口。

它是整个后端服务启动时的重要基础组件。

### 4.2 gcc：课程与认证目录服务

`gcc` 可以理解为“认证项目与课程目录中心”。

主要职责：

- 管理认证项目。
- 管理认证管线 Pipeline。
- 管理认证阶段 Stage。
- 管理课程、单元、学习路径等基础配置。
- 支持课程/认证版本发布。
- 为商城、学习、进度等服务提供基础目录数据。

这个服务偏“静态配置”和“产品目录”，决定系统里有哪些认证、课程、阶段和规则。

根据项目约定，`gcc` 是当前 MySQL 迁移到 PostgreSQL 的试点服务，并且已经完成迁移。

### 4.3 glms：学习管理服务

`glms` 是 Learning Management System，负责课程学习资源管理。

主要职责：

- 管理课程。
- 管理章节、课时、课件。
- 管理视频、PDF、文档等学习资料。
- 管理测验、题目、选项。
- 生成对象存储上传链接和访问链接。
- 跟踪或辅助学习行为。

它对接 S3、MinIO 或 OSS 等对象存储，用来保存课程资料、视频、PDF 等文件。

### 4.4 gprog：学习进度与流程引擎服务

`gprog` 是整个认证流程中非常关键的服务，可以理解为“考生进度引擎”。

主要职责：

- 为考生创建认证/课程学习实例。
- 记录每个考生在每个认证管线中的进度。
- 判断当前阶段是否完成。
- 推动考生从一个阶段进入下一个阶段。
- 处理学习完成、考试完成、重考申请等流程事件。
- 支持管理员强制调整学习或认证进度。

它通常会接收来自商城、学习、考试等服务的事件，然后推动整个认证流程继续向前。

### 4.5 gmall：商城与订单服务

`gmall` 负责课程和认证产品的购买流程。

主要职责：

- 展示可购买的认证项目或课程。
- 创建认证项目订单。
- 创建阶段订单。
- 创建课程重修或重考订单。
- 查询订单状态。
- 支付成功后触发履约流程。
- 与 `gcc`、`gcreds`、`gpay`、`gprog` 等服务协作。

典型流程：

```text
考生选择认证项目
    |
gmall 创建订单
    |
gpay 创建支付
    |
Stripe 完成支付
    |
gmall 确认支付成功
    |
gprog 创建学习/认证实例
```

### 4.6 gpay：支付服务

`gpay` 是内部支付网关服务，负责封装外部支付平台。

主要职责：

- 创建支付请求。
- 查询支付状态。
- 取消支付。
- 处理 Stripe Webhook。
- 更新支付交易状态。
- 通知订单服务支付结果。

当前项目中，Stripe 是主要外部支付集成。

### 4.7 gmbr：会员服务

`gmbr` 是 Membership Service，负责会员相关业务。

主要职责：

- 管理会员等级。
- 管理订阅状态。
- 管理账单记录。
- 处理会员生命周期事件。

它通常和支付、商城、认证资格等业务存在关联。

### 4.8 gcreds：资格与证书服务

`gcreds` 负责认证资格审核和证书发放。

主要职责：

- 管理考生资格申请。
- 管理审核流程。
- 记录审核轨迹。
- 判断考生是否具备某项资格。
- 在满足条件后发放证书。
- 支持证书查询、撤销等操作。

它属于认证业务的最终结果服务，也就是“考生是否通过认证、是否拿到证书”的核心服务。

### 4.9 gexam：考试服务

`gexam` 负责考试预约、考试代码、成绩回传等业务。

主要职责：

- 创建考试安排。
- 管理考试预约。
- 校验考试代码。
- 对接 Prometric / GEE 外部考试系统。
- 接收或拉取考试成绩。
- 处理考试结果回调。
- 将考试结果传递给进度服务或其他业务服务。

它是内部认证系统和外部考试平台之间的桥梁。

### 4.10 gmid：身份映射服务

`gmid` 负责外部身份和内部业务身份之间的映射。

主要职责：

- 将 Casdoor 用户 UUID 映射为内部业务 ULID。
- 将内部 ULID 映射回 Casdoor UUID。
- 管理考生编号、内部用户标识等身份关系。

这个服务存在的原因是：认证系统的登录身份和业务系统的用户身份通常不是同一个概念，需要有一个统一的映射层。

### 4.11 gmsg：站内消息服务

`gmsg` 负责系统内通知和站内信。

主要职责：

- 给考生发送站内消息。
- 查询消息列表。
- 标记消息已读。
- 删除消息。
- 管理消息模板。
- 支持其他服务发送或撤回消息。

例如订单状态变化、考试提醒、证书发放通知等，都可以通过这个服务触达用户。

### 4.12 gmail：邮件服务

`gmail` 负责邮件发送。

主要职责：

- 管理邮件模板。
- 创建邮件发送任务。
- 查询邮件发送状态。
- 取消邮件任务。
- 对接 SMTP 服务发送事务性邮件。

它主要用于发送注册通知、考试提醒、支付通知、证书通知等邮件。

### 4.13 gpdf：PDF 生成服务

`gpdf` 是 PDF 生成服务。

主要职责：

- 根据模板和数据生成 PDF。
- 生成证书、收据、证明文件等。
- 为 `gcreds` 等服务提供文档生成能力。

### 4.14 geemock、geewhook、geex：考试系统辅助工具

这些不是核心业务微服务，而是围绕外部 GEE / Prometric 考试系统的辅助工具。

主要作用：

- `geemock`：模拟外部考试系统，方便本地或测试环境联调。
- `geewhook`：模拟或接收考试系统 Webhook。
- `geex`：命令行工具，用于测试外部考试 API 或处理考试相关操作。

### 4.15 gstripe：Stripe 测试工具

`gstripe` 主要用于 Stripe 支付流程验证。

主要作用：

- 模拟 Stripe 客户端。
- 模拟 Stripe Webhook。
- 测试支付创建、支付成功、支付失败等流程。

## 5. 共享模块与协议

### 5.1 cftp：共享 Go 模块

`cftp` 是项目中的共享模块，主要包含：

- 公共工具包。
- 公共结构体。
- 数据库连接工具。
- 错误处理工具。
- gRPC 生成代码。
- Protobuf 源文件。

其中：

```text
cftp/protofiles/
```

保存所有服务的 `.proto` 文件。

如果修改了 `.proto` 文件，需要运行生成脚本重新生成 Go 代码：

```powershell
.\cftp\protofiles\gen.ps1
```

或者只生成某一个服务：

```powershell
.\cftp\protofiles\gen.ps1 mall
```

## 6. 数据与中间件架构

### 6.1 PostgreSQL

PostgreSQL 是系统主要关系型数据库，用于存储：

- 用户业务身份。
- 课程目录。
- 学习进度。
- 订单。
- 支付记录。
- 消息。
- 邮件任务。
- 证书。
- 审核记录。
- 考试记录。

项目正在从 MySQL 迁移到 PostgreSQL。根据项目约定，PostgreSQL 服务需要使用原生 `$1`、`$2` 这种占位符，而不是 MySQL 的 `?` 占位符。

### 6.2 Redis

Redis 主要承担两类职责：

- 缓存。
- 分布式锁。

项目约定中要求：所有数据库写操作都必须使用 Redis 分布式锁。

这样可以降低多个服务、多个实例并发写同一份业务数据时产生冲突的风险。

在条件允许时，数据库写操作还应该结合：

- SQL 行锁。
- 乐观锁。
- version 字段校验。

### 6.3 NATS JetStream

NATS JetStream 是系统的消息中间件。

主要用于：

- 服务之间的异步通信。
- 发布/订阅事件。
- 可靠消息投递。
- 订单履约事件。
- 支付结果事件。
- 学习进度事件。
- 邮件通知事件。
- 站内信通知事件。

使用消息队列的好处是，可以减少服务之间的强耦合。例如支付成功后，不一定要同步调用所有后续服务，而是可以发布事件，由相关服务订阅处理。

### 6.4 对象存储：S3 / MinIO / OSS

对象存储用于保存大文件和静态资源，例如：

- 视频课程。
- PDF 课件。
- 学习资料。
- 证书 PDF。
- 收据 PDF。

在本地或测试环境中，可以使用 MinIO 模拟 S3。

在生产环境中，可以切换为 AWS S3、阿里云 OSS 或其他兼容 S3 协议的对象存储。

### 6.5 Casdoor

Casdoor 是系统的认证中心。

主要职责：

- 用户登录。
- 用户身份认证。
- JWT Token 签发。
- 用户权限或角色管理。
- 为 BFF 提供认证集成能力。

业务系统内部不会直接把 Casdoor UUID 当成业务主键，而是通过 `gmid` 做身份映射。

## 7. 典型业务流程

### 7.1 考生购买认证项目流程

```text
考生登录系统
    |
前端请求 BFF
    |
BFF 校验 Casdoor Token
    |
BFF 调用 gmall 创建订单
    |
gmall 查询 gcc 获取认证项目信息
    |
gmall 调用 gpay 创建支付
    |
gpay 对接 Stripe
    |
考生完成支付
    |
Stripe Webhook 回调 gpay
    |
gpay 更新支付状态
    |
gmall 确认订单支付成功
    |
gprog 创建考生认证进度实例
    |
gmsg / gmail 发送通知
```

### 7.2 考生学习流程

```text
考生进入课程
    |
BFF 调用 glms 获取课程资料
    |
glms 校验学习权限
    |
glms 生成对象存储访问链接
    |
考生观看视频 / 阅读资料 / 完成测验
    |
学习进度更新
    |
gprog 推进认证阶段
```

### 7.3 考试预约与成绩回传流程

```text
考生满足考试条件
    |
gprog 判断可进入考试阶段
    |
BFF 调用 gexam 获取考试预约链接
    |
考生跳转到 Prometric / GEE 预约或考试
    |
外部考试系统回调 gexam
    |
gexam 处理考试成绩
    |
gprog 根据成绩推进流程
    |
通过后触发 gcreds 证书相关逻辑
```

### 7.4 证书发放流程

```text
考生完成学习和考试要求
    |
gprog 判断认证流程完成
    |
gcreds 校验证书发放资格
    |
gpdf 生成证书 PDF
    |
gcreds 保存证书记录
    |
gmsg / gmail 通知考生
```

## 8. 部署架构说明

项目部署文件主要放在：

```text
deployment/
```

该目录下包含 Kubernetes 部署清单，用于部署系统运行所需的服务和基础设施。

从当前目录内容看，部署架构主要包括：

- Traefik：入口网关和 HTTPS 证书处理。
- Casdoor：认证服务。
- cfgserver：配置中心。
- 各个 Go 微服务：`gcc`、`gcreds`、`gexam`、`glms`、`gmail`、`gmall`、`gmbr`、`gmid`、`gmsg`、`gpay`、`gpdf`、`gprog`。
- NATS：消息队列。
- MinIO：对象存储。
- PostgreSQL / PgBouncer 外部访问服务。
- Redis 外部访问服务。
- Loki、Grafana、Vector：日志与监控。
- Ingress：对外暴露 Casdoor、Grafana、Webhook、MinIO 等入口。

---

## 9. Kubernetes 部署分层

### 9.1 入口层

入口层主要由 Traefik 和 Ingress 组成。

Traefik 负责：

- 接收外部 HTTP/HTTPS 请求。
- 自动处理 HTTPS。
- 将请求路由到 Kubernetes 内部服务。
- 支持 HTTP 自动跳转 HTTPS。

当前 Ingress 中暴露了这些入口：

- `cftpcasdoor.llwan.top`：Casdoor 登录认证入口。
- `cftpgrafana.llwan.top`：Grafana 监控入口。
- `cftp.llwan.top/geewebhook`：考试系统 Webhook 入口，转发到 `gexam`。
- `cftp.llwan.top/gstripewebhook`：Stripe Webhook 入口，转发到 `gpay`。
- `s3cftp.llwan.top`：MinIO / S3 对象存储入口。

### 9.2 应用服务层

应用服务层就是所有 Go 微服务。

每个微服务通常都有独立的 Kubernetes Deployment 和 Service，例如：

- `gcc-deployment.yaml`
- `gmall-deployment.yaml`
- `gpay-deployment.yaml`
- `gprog-deployment.yaml`
- `gexam-deployment.yaml`

这些服务一般运行在同一个 Kubernetes namespace 中，例如当前清单中使用的是：

```text
cftp-test
```

服务之间通过 Kubernetes Service 名称进行内部访问。

### 9.3 配置层

`cfgserver` 负责集中配置下发。

部署时需要确保：

- `cfgserver` 先于业务服务可用。
- 各微服务能够访问 `cfgserver`。
- 配置文件、密钥、数据库连接、Redis、NATS 等信息正确。

当前项目中还出现了 OpenBao/Vault transit 相关初始化命令，说明配置或密钥加解密可能依赖 OpenBao/Vault 类组件。

### 9.4 数据层

数据层包括 PostgreSQL、PgBouncer、Redis。

当前 `deployment/psql-redis.yaml` 中使用的是 Kubernetes Service + Endpoints 的方式，将 Kubernetes 内部服务指向外部主机上的 PgBouncer 和 Redis。

也就是说，数据库和 Redis 不一定部署在 Kubernetes 集群内部，而是可以部署在 VPS 或外部服务器上，然后通过 Kubernetes Service 暴露给 Pod 使用。

当前清单体现的是：

```text
pgbouncer-external -> 外部 PgBouncer
redis-external -> 外部 Redis
```

这种方式的好处是：

- 数据库可以独立维护。
- Redis 可以独立维护。
- Kubernetes 内部应用只需要访问固定 Service 名称。
- 后续迁移数据库位置时，可以只改 Endpoints，不一定改业务服务配置。

### 9.5 消息层

NATS 部署在 Kubernetes 中。

它负责服务间异步事件通信，并配置了持久化存储 PVC。

部署上需要关注：

- NATS 服务端口 `4222`。
- NATS 监控端口 `8222`。
- JetStream 存储目录。
- PVC 是否正常挂载。
- 服务重启后消息是否能恢复。

### 9.6 对象存储层

MinIO 用于提供兼容 S3 的对象存储能力。

主要用途：

- 本地或测试环境模拟 AWS S3。
- 保存课程资料。
- 保存 PDF。
- 保存视频。
- 为学习服务 `glms` 提供上传和查看链接。

生产环境可以根据需要切换到 AWS S3 或阿里云 OSS。

### 9.7 日志与监控层

当前部署目录中包含：

- Loki
- Grafana
- Vector

它们共同组成日志采集与可视化体系。

大致职责是：

- Vector：采集 Pod 日志。
- Loki：存储和查询日志。
- Grafana：展示日志和监控面板。

这样可以方便排查微服务运行问题，例如订单失败、支付回调失败、考试成绩同步失败等。

## 10. 部署流程概览

### 10.1 准备基础环境

需要先准备：

- Kubernetes 或 K3s 集群。
- Docker 镜像仓库。
- PostgreSQL 或 PgBouncer。
- Redis。
- 域名。
- TLS 证书或 ACME 自动签发能力。
- 对象存储，例如 MinIO、S3 或 OSS。
- 外部服务账号，例如 Stripe、SMTP、Casdoor、Prometric/GEE。

### 10.2 部署入口网关

部署 Traefik，并配置 HTTPS。

当前项目中 Traefik 配置使用了 ACME DNS Challenge，并通过 AliDNS 完成域名验证。

也就是说，证书可以自动签发和续期。

### 10.3 部署基础设施

基础设施包括：

- NATS JetStream
- MinIO
- Loki
- Grafana
- Vector
- PostgreSQL/PgBouncer 外部服务映射
- Redis 外部服务映射

这些组件需要先于业务服务启动，因为业务服务依赖数据库、Redis、消息队列和配置中心。

### 10.4 部署认证和配置服务

接着部署：

- Casdoor
- cfgserver

Casdoor 负责用户认证，cfgserver 负责配置下发。

这两个服务是后续业务服务启动和前端登录的基础。

### 10.5 部署业务微服务

然后部署所有业务微服务：

- gcc
- glms
- gprog
- gmall
- gpay
- gmbr
- gmid
- gmsg
- gmail
- gpdf
- gexam
- gcreds

部署时需要确认：

- 配置中心连接正常。
- 数据库连接正常。
- Redis 连接正常。
- NATS 连接正常。
- gRPC 服务端口正常。
- 服务之间的 DNS 名称可解析。

### 10.6 配置 Ingress

最后配置对外入口：

- Casdoor 登录入口。
- Grafana 监控入口。
- Stripe Webhook 入口。
- GEE/Prometric Webhook 入口。
- MinIO/S3 入口。

其中 Webhook 入口非常关键，因为它们是外部系统回调内部系统的通道。

## 11. 对外讲解口径

如果需要给别人快速介绍这个系统，可以这样讲：

> CFTP 是一个面向认证培训业务的微服务系统。前端分为考生端和管理端，所有请求先进入 BFF 网关，BFF 负责鉴权和接口聚合。后端微服务分别处理课程、学习、进度、订单、支付、考试、证书、消息和邮件。服务之间主要通过 gRPC 通信，异步事件通过 NATS JetStream 传递。系统使用 PostgreSQL 保存业务数据，Redis 做缓存和分布式锁，对象存储保存课程资料和 PDF 文件。认证由 Casdoor 提供，支付对接 Stripe，考试对接 Prometric/GEE。部署上使用 Kubernetes/K3s，Traefik 负责 HTTPS 入口，Grafana/Loki/Vector 负责日志和监控。

更短一点可以这样说：

> 这是一个 Go 微服务架构的认证培训平台。BFF 负责接入前端和鉴权，后端微服务分别处理课程、学习、进度、订单、支付、考试、证书、消息和邮件。系统底层依赖 PostgreSQL、Redis、NATS 和对象存储，并通过 Kubernetes 部署，Traefik 暴露外部入口，Casdoor 提供统一认证。
