针对在项目中落地的最佳实践，我为您汇总并梳理出最新、最完整的**“高性能双向 Keyset 分页（List & Count）接口设计及 SQL 规范标准”**。

在所有微服务开发中，所有分页查询（List）及配套的统计计数（Count）接口，都应严格遵循以下规范：

---

### 一、 接口设计规范 (API Protocol Standards)

#### 1. List 接口标准
* **请求参数**：
  * **必须**包含 `string cursor`（不透明游标，前端透明透传）。支持基于动作感知（Action-Aware）的编码，区分前进（Forward）与后向（Backward）。
  * **必须**包含 `uint32 page_size`（限制在安全区间内，默认 20，最大 100）。
  * **必须**支持 `SortOrder sort_order`，用来指定遍历的主序方向（默认为 `SORT_ORDER_DESC`，即从大到小/最新优先）。
* **响应参数**：
  * **必须**包含 `bool has_more`（标识当前方向上是否还有更多数据）。
  * **必须**包含 `string next_cursor`（下一页游标）。若无下一页数据，则返回空字符串。
  * **必须**包含 `string prev_cursor`（上一页游标）。若无上一页数据，则返回空字符串。
  * **禁止**在 List 接口中直接返回全局总计数（`total`），以避免深分页触发大表全表扫描。

#### 2. Count 接口标准
* **对称性原则**：每个分页 List 接口**必须**配置一个过滤条件完全一致的 Count 接口。
* **请求参数**：
  * **必须**支持 `string cursor`，以便客户端可以统计从当前锚点位置“之前/之后”的条数，提供动作感知能力。
  * **必须**包含 `uint32 limit`（最大扫描上限阈值，默认 1000，最大 10000），防止扫描整表。
  * **必须**支持 `SortOrder sort_order`，用以与配套的 List 接口保持完全相同的过滤与主序迭代方向。
* **响应参数**：
  * **必须**包含 `uint32 count`（在当前扫描上限范围内实际统计得到的行数）。
  * **必须**包含 `string next_cursor`（本次 Count 计数的**终点行游标**），以便前端或 BFF 能够基于该游标分批接续统计。
  * **必须**包含 `string prev_cursor`（本次 Count 计数的**起点行游标**）。

#### 3. 双向迭代与排序规范 (Bidirectional Iteration Standards)
* **枚举定义**：
  所有的 Proto 文件中，若包含分页/统计列表接口，**必须**统一定义如下枚举：
  ```protobuf
  enum SortOrder {
    SORT_ORDER_DESC = 0; // 默认值：自增 ID 从大到小 (最新提交优先)
    SORT_ORDER_ASC  = 1; // 自增 ID 从小到大 (最早提交优先，适用于顺序扫描或对账)
  }
  ```
* **默认值处理**：客户端不传该参数时，服务端逻辑默认将其视为 `SORT_ORDER_DESC`。

#### 4. 过滤参数统一封装 (Encapsulated Filters Message)
* **DRY 原则**：List 和 Count 接口**必须**将所有的公共过滤字段（如 operator_id、status、candidate_ulid、start_time 等）抽离并封装成一个独立的 Protobuf 消息结构（例如 `xxxFilters`）。
* **使用方式**：
  * 在 `ListxxxRequest` 与 `GetxxxCountRequest` 中，以一个单独的 `filters` 属性字段嵌套引入该过滤结构：
    ```protobuf
    message ApplicationFilters {
      string candidate_ulid = 1;
      string cred_def_ulid = 2;
      repeated string statuses = 3;
    }

    message ListApplicationsRequest {
      ApplicationFilters filters = 1;
      string cursor = 2;
      uint32 page_size = 3;
      SortOrder sort_order = 4;
    }

    message GetApplicationCountRequest {
      ApplicationFilters filters = 1;
      uint32 limit = 2;
      string cursor = 3;
      SortOrder sort_order = 4;
    }
    ```

### 二、 动作感知游标设计 (Action-Aware Cursor)

为了支持“上一页”和“下一页”的双向翻页逻辑，而不破坏无状态、高性能的设计：
1. **游标格式**：
   游标底层封装了逻辑锚点 ID（如自增主键 `id`）以及本次翻页动作的方向。格式为：
   * 前进翻页：`f:{ID}`
   * 后退翻页：`b:{ID}`
2. **安全编码**：将上述格式使用 Raw URL-Safe Base64 进行编码，隐藏底层数据库物理 ID。
3. **兼容性**：若传入的游标是旧版纯数字字符串，服务端将默认降级处理为前向翻页模式（`f:{ID}`）。

### 三、 服务端处理与 SQL 查询规范 (Backend & SQL Standards)

#### 1. 双向定位 SQL 拼接与内存反转 (List 逻辑)
List 接口使用 Seek 方式查询。执行逻辑步骤如下：

##### 步骤 1：解析参数
通过公共函数 `ValidateListPageWithAction(cursor, pageSize)` 获取底层锚点 ID（`lastID`）、翻页方向（`isBackward`）及限制值（`limit`）。

##### 步骤 2：确定 SQL 排序方向 (Pagination Sort Direction)
SQL 的排序方向由客户端请求的主序方向（`defaultSortAsc`）与本次翻页的方向（`isBackward`）共同决定：
```go
defaultSortAsc := req.GetSortOrder() == pb.SortOrder_SORT_ORDER_ASC
paginationSortAsc := defaultSortAsc != isBackward
```
* **当向后翻页 (`isBackward = true`) 时，必须翻转 SQL 的物理排序方向**。例如：默认 DESC 排序下，向后翻页需要以 ASC 方式向上拉取数据。

##### 步骤 3：数据库 Seek 查询 (`limit + 1` 条记录)
根据计算出的物理排序方向 `paginationSortAsc`，在 Repository 层执行对应的 Seek 查询：
* **`paginationSortAsc == false` (DESC 物理排序)**:
  ```sql
  SELECT columns FROM table 
   WHERE filters AND id < $last_id 
   ORDER BY id DESC LIMIT $limit + 1;
  ```
* **`paginationSortAsc == true` (ASC 物理排序)**:
  ```sql
  SELECT columns FROM table 
   WHERE filters AND id > $last_id 
   ORDER BY id ASC LIMIT $limit + 1;
  ```

##### 步骤 4：切片判定与内存反转
* 调用 `util.PaginateSlice(list, limit)` 裁剪掉额外查询的第 `limit+1` 条记录，并得到 `has_more` 标识。
* **重要**：如果 `isBackward` 为 `true`，**必须**在内存中调用 `util.ReverseSlice(items)` 将返回的记录反转。这保证了即使底层数据库查询为了双向 Seek 翻转了排序，最终输出给前端的列表顺序依然与客户端请求的主序方向一致。

##### 步骤 5：生成双向游标
* 下一页游标：`nextCursor := util.EncodeNextCursor(list, hasMore, isBackward, getID)`
* 上一页游标：`prevCursor := util.EncodePrevCursor(list, req.Cursor, hasMore, isBackward, getID)`

---

#### 2. Capped Threshold 双向计数逻辑 (Count 逻辑)
为防范在大表上运行全表 `COUNT(1)`，Count 的 SQL 编写必须使用**限制行的子查询**，且完全对齐双向 Seek 定位：

##### 步骤 1：解析参数并翻转排序方向
* 通过 `ValidateCountLimitWithAction(cursor, limit)` 解析出 `lastID`，`isBackward`，以及 `limit`。
* 同样计算出底层的物理排序方向 `paginationSortAsc := defaultSortAsc != isBackward`。

##### 步骤 2：子查询 Capped SQL 统计
* **`paginationSortAsc == false` (DESC 物理排序)**:
  ```sql
  -- 使用带有 LIMIT 的子查询只扫描阈值内的行数，并查出统计数与批次终点 ID (MIN)
  SELECT COUNT(1) as cnt, COALESCE(MIN(id), 0) as min_id 
    FROM (
        SELECT id 
          FROM table 
         WHERE filters AND id < $last_id 
         ORDER BY id DESC 
         LIMIT $limit
    ) t;
  ```
* **`paginationSortAsc == true` (ASC 物理排序)**:
  ```sql
  -- 使用带有 LIMIT 的子查询，并查出批次终点 ID 的最大值 (MAX) 作为下一批统计锚点
  SELECT COUNT(1) as cnt, COALESCE(MAX(id), 0) as min_id 
    FROM (
        SELECT id 
          FROM table 
         WHERE filters AND id > $last_id 
         ORDER BY id ASC 
         LIMIT $limit
    ) t;
  ```

##### 步骤 3：返回双向游标
* 计数的终点游标（Next）：`nextCursor := util.EncodeCountNextCursor(res, limit, lastID, isBackward)`
* 计数的起点游标（Prev）：`prevCursor := util.EncodeCountPrevCursor(res, limit, req.Cursor, lastID, isBackward)`

### 四、 数据库表与索引设计规范 (Database Schema & Indexing Standards)

为了使上述 Seek 过滤和排序能够完美走索引的快速扫描（Index Only Scan）而不需要内存排序，SQL 表的索引必须按以下原则建立：

#### 1. 组合索引设计原则
当过滤条件包含字段 `A`，且结果按主键 `id` 排序时，必须建立包含 `id` 的联合索引：
* **核心模式**：`(FilterColumns..., id DESC)`
* **示例**：
  * 若基于 `cred_def_id` 过滤，查询 `ORDER BY id DESC`：
    👉 索引定义：`CREATE INDEX ON cred_applications (cred_def_id, id DESC);`
  * 若同时基于 `candidate_id` 与 `cred_def_id` 过滤，且排序 `ORDER BY id DESC`：
    👉 索引定义：`CREATE INDEX ON cred_applications (candidate_id, cred_def_id, id DESC);`

#### 2. 避免冗余与无效索引
* 原先基于 `created_at DESC` 进行分页排序的索引，在改为基于 `id` 游标分页后，**必须全部更新为 `id DESC`**。
* 不要为不需要单独做降序分页过滤的字段建单独的联合降序索引。

### 五、 其他补充保障规范 (Extra Standards)

1. **游标的不透明性**：在底层，游标是 `int64` 物理主键；在接口层，游标是 Base64 编码的 `string`，保证外界无法自行拼装甚至感知底层字段逻辑。
2. **时间 UTC 强制对齐**：所有通过 API 输出的日期和时间，必须使用 `.UTC().Format(time.RFC3339)` 强制统一输出，避免由于时区差异导致的数据统计偏移。
